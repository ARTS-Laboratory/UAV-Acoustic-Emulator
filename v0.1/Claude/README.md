# MATLAB UAV Acoustic Emulator

A modular MATLAB toolbox that simulates microphone time-series signals
generated by a UAV flying through 3D space above a 2D array of ground
microphones. It is a **simulator**, not a detection algorithm — the
goal is physically plausible synthetic data for downstream work such as
TDOA localization, classification, or distributed-sensing experiments.

Realism target: about 80% — enough that signals change with range,
channels differ from each other, motion affects timing/frequency, and
noise can obscure the source — without over-engineering the acoustics.

## Files

| File | Purpose |
|---|---|
| `simulate_uav_acoustics.m` | Main entry point. Validates inputs, orchestrates the pipeline, packages outputs. |
| `generate_uav_trajectory.m` | Builds a 3D UAV path: straight line, arc, loiter circle, or piecewise waypoints. |
| `generate_source_signal.m` | Builds the source waveform: tone, multitone, rotor-like harmonic stack, noise burst, or custom. |
| `propagate_to_microphones.m` | Computes per-microphone distance, delay, attenuation, optional Doppler, optional ground reflection. |
| `add_noise.m` | Adds white / pink / wind-like noise at a target SNR. |
| `save_simulation.m` | Saves `Y`, `t`, `meta`, and `extras` to a `.mat` file. |
| `plot_simulation_results.m` | Plots trajectory + mic layout, altitude, sample waveforms, and a spectrogram. |
| `demo_uav_acoustics.m` | Three worked examples (5-mic straight pass, 20-mic arc with reflection, loiter with noise burst). |

## Quick start

```matlab
addpath('/path/to/uav-acoustic-emulator');

micXY = [0 0; 50 0; 0 50; -50 0; 0 -50];   % 5 mics, meters
t     = (0:1/4000:10)';                    % 10 s @ 4 kHz

uavXYZ = generate_uav_trajectory(t, 'straight', ...
    'startXY', [-200 0], 'endXY', [200 0], 'altitude', 80);

[Y, t, meta, extras] = simulate_uav_acoustics(micXY, uavXYZ, t, ...
    'sourceType', 'rotor', 'sourceFreq', 120, ...
    'snrDb', 12, 'enableDoppler', true);

save_simulation('run_001.mat', Y, t, meta, extras);
plot_simulation_results(Y, t, meta);
```

`Y` is `numSamples x numMics`; `Y(:, i)` is the signal at microphone `i`.

## Signal model

For microphone `i`:

```
y_i(t) = (1 / r_i(t)^alpha) * s(t - tau_i(t)) + n_i(t)
```

- `r_i(t)` = distance from the UAV to microphone `i` (mic assumed at `z = 0`)
- `alpha`  = attenuation exponent (1 = inverse distance, 2 = inverse square)
- `tau_i(t) = r_i(t) / c`
- `s(t)`   = source waveform
- `n_i(t)` = additive noise at a controllable SNR

## Doppler design note

A continuously time-varying delay applied via interpolation already
reproduces the waveform compression/stretching that *is* Doppler shift
— no separate model is needed to get the physics right. To keep
`'enableDoppler'` a meaningful on/off switch rather than a no-op:

- **`enableDoppler = true`**: delay is applied via full-resolution
  interpolation at the exact retarded time, so motion-induced pitch
  shift comes through naturally.
- **`enableDoppler = false`**: delay is applied piecewise-constant,
  updated roughly every 20 ms. Timing still tracks range (important
  for TDOA-style work) but no continuous pitch shift is introduced.

A Doppler shift *estimate* (from radial velocity) is always returned
in `extras.doppler` for diagnostics, regardless of the flag.

## Reflection design note

When `enableReflection = true`, a second, geometrically distinct
"image-source" arrival is added using a small assumed microphone
capsule height above the ground (so the reflected path length differs
meaningfully from the direct one). This is a simplification, not a
full acoustic boundary/reflection-coefficient model — by design, per
the "don't overcomplicate the physics" guidance.

## Inputs at a glance

- `micXY` — `N x 2`, any `N` (tested with 5 and 20 mics)
- `uavXYZ` — `T x 3`, paired with time vector `t`
- Acoustic: `fs`, `c` (default 343 m/s), `alpha` (default 1), `snrDb`
  (default 15 dB), `enableDoppler`, `enableReflection`
- Source: `sourceType` (`tone`/`multitone`/`rotor`/`noiseburst`/`custom`),
  `sourceFreq`, `sourceWaveform` (for `custom`)
- Environment: `noiseType` (`white`/`pink`/`wind`), `dropoutRate`
  (probability a mic channel is dead)
- `seed` for reproducibility

## Outputs

- `Y` — noisy signal matrix
- `t` — time vector used
- `meta` — struct: mic coords, UAV trajectory, source/attenuation/noise
  settings, enabled effects, dropout mask, seed, timestamp
- `extras` (optional 4th output) — `cleanY`, `distances`, `delays`,
  `doppler` (diagnostic estimate)

## Validation

`simulate_uav_acoustics.m` checks: numeric/finite mic and UAV
coordinates, matching trajectory/time lengths, monotonically increasing
time, positive sampling rate, valid numeric SNR, and dropout rate in
`[0,1]`, before any simulation work begins.

## Notes

- No toolboxes are strictly required; `plot_simulation_results.m` will
  skip the spectrogram panel gracefully if Signal Processing Toolbox's
  `spectrogram` is unavailable.
- Everything is vectorized per-microphone-channel and loops only over
  microphones (typically 5–20), so it scales comfortably to larger
  arrays and longer durations.
- This toolbox intentionally does **not** implement detection,
  localization, or classification — it only generates the synthetic
  channel data those algorithms would consume.
